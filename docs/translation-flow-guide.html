i18n Sync Translations: Complete Flow + Context Inference

This document explains what happens at every stage of translation sync, from missing key detection to final file merge. It is written for non-technical users, with technical depth where it matters.

Core idea: the extension translates only missing keys, uses nearby sibling keys as context, validates model output, retries safely, and merges results without overwriting existing translated entries.

1) End-to-End Process Map

flowchart TD A[Sync Trigger\nCommand / Context Menu / Auto-Sync] --> B[Detect i18n format] B --> C[Read English source] C --> D[Find missing keys per language] D --> E{Any missing keys?} E -- No --> F[Stop: everything in sync] E -- Yes --> G[Create batches by language] G --> H[Infer context for each key] H --> I[Build numbered prompt with commented context] I --> J[Cursor CLI translation call] J --> K[Parse and validate output] K --> L{Valid and complete?} L -- No --> M[Retry with backoff / fail-safe] M --> J L -- Yes --> N[Merge into target files] N --> O[Summary and status update]

Human-friendly flow

  1. Start sync manually (or auto-sync triggers after save debounce).
  2. Tool checks if project uses JSON files or Java properties files.
  3. English keys are compared against each language file.
  4. Only missing keys are queued for translation.
  5. Context is gathered from neighboring sibling keys.
  6. Cursor CLI returns numbered translated lines.
  7. Results are verified and merged back into files.

Where quality comes from

  • Context window: nearby same-prefix keys improve terminology consistency.
  • Existing translations first: if available, existing target values are used as context.
  • Strict output parser: expects N. translated value lines only.
  • Safe merge: existing translated keys are not replaced in missing-key mode.
  • Retry system: temporary CLI/model failures recover automatically.

Key settings: contextWindowSize, batchSize, concurrentLimit, maxRetries.

2) Context Inference - Technical Deep Dive

How keys are grouped

A key prefix is all segments before the last dot. Example:

settings.agreements.select_file
prefix = settings.agreements

Context uses sibling keys with the same prefix to stay domain-consistent.

How context is selected

  • Scan ordered English key list for nearest siblings (before/after).
  • Take up to contextWindowSize on each side.
  • Exclude keys currently being translated in this batch.
  • Use target-language value if present, otherwise fallback to English.

Long locale fallback: when translating a locale like pt-BR, if that file has no useful context yet, the engine can load pt as context fallback (for consistency), while still generating output for pt-BR.

3) Prompt Shape (what the model receives)

Context is added as comments and must not be translated. Actual work items are numbered lines:

# --- Section: settings.agreements ---
# Context (already translated siblings for reference, DO NOT translate these):
#   settings.agreements.title: Vereinbarungen
#   settings.agreements.download_document: Dokument herunterladen
#   ...
# Translate the following:
1. settings.agreements.select_file: "Select file"
2. settings.agreements.uploading: "Uploading file..."

Expected output format: 1. ..., 2. ... only, no extra prose.

5) Reliability and Guardrails

  • CLI path is validated before execution.
  • Translation call timeout prevents hangs.
  • Response count mismatch is treated as invalid output.
  • Exponential retries for recoverable failures.
  • Fatal quota errors stop cleanly with user-visible message.
  • Cancellation and resumable progress are supported.

6) Suggested Team Operating Model

  • Use consistent translationTone across teams/projects.
  • Keep contextWindowSize between 3 and 7 for most products.
  • Review legal/compliance copy manually even after automation.
  • Use auto-sync for day-to-day velocity, manual sync for release QA checks.
  • Audit glossary-heavy areas (billing, tax, contracts) periodically.
If terminology drifts, first adjust context quality and source key organization before changing model settings.
